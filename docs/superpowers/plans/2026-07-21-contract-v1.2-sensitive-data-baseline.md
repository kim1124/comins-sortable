# Comins Sortable Contract v1.2 Sensitive Data Baseline Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Apply the Comins Contract v1.2 sensitive-data prevention baseline to `comins-sortable` without defining its product API, public npm package, or release workflow.

**Architecture:** Use a private, zero-dependency Node tooling manifest to run a repository-owned copy of the audited governance scanner. Versioned Git hooks, required GitHub Actions, and checksum-pinned Gitleaks protect Git surfaces now; a committed first-release checklist keeps exact package and npm identity gates mandatory when product packaging is separately approved.

**Tech Stack:** Markdown, Node.js 24 ESM, Node test runner, Git hooks, GitHub Actions, Gitleaks 8.30.1, npm lockfile with no third-party packages.

## Global Constraints

- Start only after sibling `../governance` completes `2026-07-21-contract-v1.2-sensitive-data-standard.md` locally.
- Keep `comins-sortable` an independent Git repository and CI unit.
- Do not choose the sortable engine, React API, package name, npm scope, dependencies, exports, browser matrix, version, or release workflow in this work.
- Keep the tooling manifest `private: true`; do not run `npm publish`, `npm stage publish`, or create a product tarball.
- Add no npm production or development dependency.
- Never commit an actual personal name, personal email, identifying local path, credential, key, token, private key, or secret-derived fingerprint as a fixture or allowlist value.
- Failure output contains only a rule ID, non-sensitive repository-relative path or per-run ordinal redacted label, and line number.
- Fail closed on unreadable input, malformed Git ranges, symlink traversal, unsupported encoding or archive entry, and checksum mismatch.
- Do not push, open or update a PR, change GitHub settings, publish, or perform another external write without a separate explicit request.

```markdown
# First Release Security Checklist

This repository is not ready to publish a product package. Complete these items only in a separately approved package-scope plan.

- Confirm the public npm name, package owner handle, API, runtime, peers, exports, browser support, and release version.
- Replace the private tooling manifest with the approved product manifest without weakening the security scripts.
- Preserve immutable install, unit, build, browser, and consumer-install gates.
- Add an unprivileged verify-and-pack job and create exactly one npm `.tgz`.
- Run `security-audit.mjs package` against that exact archive.
- Safely extract the same archive and run checksum-pinned Gitleaks directory mode.
- Upload only the archive that passed both package scans.
- Verify Git author and committer identities and authenticated npm profile email use approved public identities without printing them.
- Bootstrap the new npm package interactively with maintainer 2FA and no automation token.
- After bootstrap, use the approved trusted publisher and environment-gated staged publish flow.
- Verify registry package metadata after publication without printing personal values.
```
